## The 5 sub-projects

Originally proposed as one request, decomposed per brainstorming scope-check because answers in one sub-project gate decisions in others.

### A. Hierarchy tree in Mission Control (UI/viz)

Visual org chart of the team. **Blocked on B + C** — the current data model has no reporting lines (Flint routes to agents but isn't their boss). A tree requires hierarchy semantics that don't exist yet.

Earns its place only if subagents (C) make the hierarchy meaningful. Otherwise it's visual sugar over a flat roster (YAGNI).

Estimated scope: ~0.5-1 session, co-designed with C.

### B. New agents — the roster expansion [STARTING HERE]

Hire specialists to close capability gaps the current 9-agent team keeps flagging. Candidate set I pitched:

- **CPO / Product Lead** — roadmap, OKRs, prioritization (distinct from Flint's ideation lens)
- **CTO** — engineering leadership, team structure, tech debt strategy (distinct from Atlas's solo architect lens)
- **Data Lead** — metrics, experimentation, dashboards (distinct from Vex's market intel)
- **Legal / Compliance** — privacy, T&Cs, Saudi PDPL, GDPR
- **CFO / Finance** — pricing, unit economics, fundraising

User asked for CPO + CTO explicitly. Other three are my recommendations based on gaps seen in past project runs (pet healthcare flagged monetization, digital signature flagged legal, Saudi neobank flagged unit economics).

Estimated scope: ~1-2 sessions. Sub-project going through the full brainstorming → spec → plan → implementation cycle now.

### C. Subagent delegation (architectural change — the biggest)

Each main agent can spawn specialist subagents for specific tasks. Big architectural decision with three axes:

1. **Lifetime** — ephemeral (die after task), persistent (named, reusable), or template-based (spawn from template with parameters)
2. **Scope of a "task"** — phase? deliverable? tool invocation? anything?
3. **Ownership** — subagents as children of main (inherit persona) or independent specialists (separate persona)?

Wrong answers here = combinatorial explosion of 47 agents by month 3, OR a feature that's never used. Middle ground critical. **Template-based + ephemeral default + occasional persistent-for-recurring-tasks** is my current bet.

Estimated scope: ~2-3 sessions. Must co-design with A since A visualizes C.

### D. Perfect context across business/project (memory system)

"Perfect context" is aspirational — Claude's context window is finite. Real design question: what does "the business" mean?

- **Project dossier** — one file per project, shared context, updated each phase
- **Knowledge graph** — entities + relationships (products, competitors, decisions, experiments)
- **Episodic memory** — time-ordered log, queryable by agent

Overlaps significantly with "Memory & Calibration (A+D)" from the v4.0 roadmap, but user's framing is project-knowledge-base, not track-record-calibration.

Estimated scope: ~2-3 sessions. Last, benefits most from B + C + A being in place.

### E. Tools & platforms catalog (new Mission Control tab)

Not just a list — a static list rots the day it ships. Must have teeth:
- Per-agent "which tools this agent should use" assignments
- Usage indicator (are you actually using Figma?)
- Gap analysis ("you have Vex but no Similarweb/Ahrefs")
- Cost vs. value ($/month per tool, agents using it)
- Recommendations ("You're paying for Notion + Linear + Jira — pick one")

User said "assess and optimize" → this includes an audit of their actual stack as part of the feature.

Estimated scope: ~1-2 sessions. Independent of A/B/C/D, can run in parallel.

4. **D (Context system)** — last, richest, benefits from all the above

## Landmines to avoid (across all 5 sub-projects)

1. **Tree for tree's sake** — if it's visual sugar over flat roster, YAGNI
2. **Over-persistent subagents** — every task spawning named lifers → 47 agents by month 3
3. **Hire-everyone trap** — adding 10 agents makes The Forge louder, not smarter. Curate to 3-5.
4. **Context drift** — "perfect context" must negotiate with context-window limits
5. **Tool list as museum piece** — needs audit loop + alarm for unused-but-paid
6. **Meta-protocol scaling** — every new agent adds routing complexity; at 14+ agents the protocol needs rethinking (is Flint still sole orchestrator?)

## Success criteria (across all sub-projects)

- Team curated, not bloated (roster ≤ 14 total at any time)
- Every feature "earns its place" with at least one real user interaction it unlocks
- No new feature loses context continuity with v3.1 Evidence Pipes
- Every sub-project ends with a live validation run (Task 14-style)
- Each sub-project ships independently and is mergeable on its own

## Open questions carried across sub-projects

- **Department creation:** does CPO get its own Product department, or join Strategy? (Sub-project B decision.)
- **Subagent UI:** are subagents shown in the pixel office? Tree? Dashboard only? (C + A decision.)
