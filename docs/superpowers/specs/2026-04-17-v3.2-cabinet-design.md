# The Forge v3.2 Cabinet — Design Spec

**Date:** 2026-04-17
**Status:** Approved for implementation planning
**Scope:** Sub-project B of the v3.2 expansion roadmap — 6 new agents (CPO, CTO, CFO, CMO, Legal, Data Lead), C-Suite + IC two-tier structure, Protocol v3.1 → v3.2, 6 elite enhancements
**Sequel sub-projects:** A (hierarchy tree), C (subagent delegation), D (context system), E (tools catalog) — see `2026-04-17-v3.2-expansion-roadmap.md`

---

## Purpose

Evolve The Forge from a flat 9-agent peer strike team into a **two-tier org** with a real C-Suite that does executive judgment and an IC cohort that does specialist execution. Add 6 agents filling the biggest capability gaps seen in v3.0/v3.1 project runs (monetization, product management, engineering leadership, legal, marketing strategy, data). Bake in 6 elite-tier enhancements: named practitioner playbooks, pre-mortems, decision logs with reversibility, function-vs-function rivalries, signature artifacts, and an explicit escalation ladder.

## Competition class

Not "bigger team." The comparison is **real digital product C-Suites at high-functioning growth-stage companies**:
- Stripe, Figma, Linear, Vercel, Notion — product-led orgs with clear CPO/CTO/CFO functions
- Amazon — decision-log + pre-mortem + Type 1/Type 2 reversibility culture
- Bridgewater — radical transparency + believability-weighted debate (extended from v3.0 already)
- Team Topologies (Skelton & Pais) — engineering org as explicit structure, not accident

Current Forge state: has the protocol discipline, missing the executive judgment layer. This closes that gap.

### Reporting relationships

- C-Suite: peer group, no internal hierarchy. Flint has ceremonial first-seat (framing authority) but cannot override peers unilaterally.
- ICs: each IC has one owning C-Suite exec via `reports_to` field. Exec owns arbitration for their function.
- User (CEO/Board): sits above the C-Suite. Final authority on deadlocks that escalate from Cabinet Floor.

### Flow

```
User brief
   ↓
Phase 1: INTAKE (Flint triages — simple query, Q&A, or project?)
   ↓
Phase 1.5: CABINET FRAMING  ← NEW
   • 5 lenses (5-min each): CSO/CPO/CTO/CFO/CMO diagnose
   • Pre-Mortem: Klein-style failure-mode elicitation
   • Output: Framing Brief + top 5 risks with owners
   ↓
Phase 2: EVIDENCE PIPES (existing v3.1 — unchanged)
   ↓
Phase 3: WAR ROOM — TWO FLOORS  ← NEW shape
   • IC Floor: specialist debate (existing, now includes Lex/Zeta)
   • Cabinet Floor: cross-functional arbitration
   • Escalation ladder: IC → owning exec → Cabinet → User
   ↓
Phase 4: ARCHITECTURE (now: Helix → Atlas → Ren → Sable)
   ↓
Phase 5: GO-TO-MARKET (now: Dune → Talon → Kira → Nyx)
   ↓
Phase 6: CHALLENGE ROUND (hedges the Phase 1.5 pre-mortem risks)
   ↓
Phase 7: FINAL DELIVERY  ← NEW format
   • Cabinet Verdict: decision + decider + dissent + reversibility
   • 5 signature artifacts (one per C-Suite exec)
   • Decision Log entries written
   • Pre-Mortem risks status
   • IC deliverables + Sources Appendix (existing)
   ↓
Phase 8: POST-MORTEM + Decision Log close
```

## Protocol v3.2 — phase-by-phase detail

### Phase 1: Intake (Flint triages) — expanded scope

Flint receives input and classifies:

| Input shape | Triggers Cabinet? | Phases run |
|---|---|---|
| "show office" / "team roster" / "hire X" | ❌ | Direct response |
| Q&A or research question | ❌ | Phase 2 Evidence Pipes only |
| Project brief | ✅ | Full Phase 1 → 8 |
| "continue project proj-X" | ✅ (review-only) | Cabinet reads Decision Log + evidence, then resumes |
| `auto-approve` meta-command | ✅ (compressed) | Framing + checkpoints auto-pass |

### Phase 1.5: Cabinet Framing (NEW)

**Activation:** project briefs only.

**Step 1 — 5 lenses (5 minutes each, ≤3 sentences):**

| Lens | Exec | Question |
|---|---|---|
| Strategic Kernel | Flint | "What's the real question? What's the diagnosis?" |
| Product shape | Cade | "Who's the user? What's the outcome? What's NOT in scope?" |
| Build class | Helix | "Greenfield or retrofit? Buy, build, or partner?" |
| Economic shape | Prism | "Unit economics shape? Break-even at what scale?" |
| Market bet | Dune | "Positioning? Who are we fighting for shelf space?" |

**Step 2 — Pre-Mortem (Gary Klein):**

> *"It's 18 months from now. We shipped this. It's a disaster. What went wrong?"*

Each C-Suite exec lists top 2 failure modes from their lens. Cabinet ranks by likelihood × impact (5×5), keeps top 5, assigns owner.

**Output stored in `forge-tasks.json`:**
```json
{
  "cabinet_framing": {
    "framing_brief": "...",
    "lenses": {
      "strategic_kernel": "...",
      "product_shape": "...",
      "build_class": "...",
      "economic_shape": "...",
      "market_bet": "..."
    }
  },
  "pre_mortem": [
    {
      "failure_mode": "Saudi PDPL non-compliance forces shutdown at 10K users",
      "likelihood": 4, "impact": 5, "score": 20,
      "owner_agent": "agent-lexx",
      "mitigation_phase": "phase_5_gtm"
    }
  ]
}
```

### Phase 2: Evidence Pipes (unchanged from v3.1)

The 8 Evidence Pipes agents (Vex, Nyx, Echo, Talon, Prism, Dune, Lex, Zeta) fan out per existing v3.1 spec. `score_agent_relevance` extended to include the 4 new pipes agents. Cade doesn't pipe — Cade consumes Evidence. Helix doesn't pipe — Helix reasons.

### Phase 3: War Room — TWO FLOORS

**IC Floor** (existing, expanded):
- Trigger: numeric-divergent Evidence conflicts (existing detector)
- Participants: ICs with Evidence on the disputed topic
- Max 2 rounds
- Resolution: scope > tier > recency (existing)

**Cabinet Floor** (NEW):
- Trigger: cross-functional trade-off (e.g., Product timeline vs Engineering capacity vs Finance runway)
- Participants: 2+ C-Suite in tension
- Max 2 rounds — ammunition is each exec's signature artifact
- Resolution: majority vote + dissent logged in Decision Log

**Escalation ladder (Standing Rule 10 — new):**

```
IC deadlock (IC Floor, 2 rounds)
   ↓
Escalates to owning C-Suite exec
   ↓ (if exec takes a side: decided; if can't:)
Cabinet Floor arbitration
   ↓ (if majority: decided + dissent logged; if deadlock:)
User (CEO/Board) decides
   ↓
All decisions logged in forge-decisions.json with full context
```

### Phase 4-6 — small changes

- **Phase 4** (Architecture): Helix kicks off with the Technology Strategy Memo's make-vs-buy framing; Atlas fills C4 detail; Ren → Sable sequence preserved
- **Phase 5** (Go-to-Market): Dune opens with the Positioning Document; Talon runs tactical AARRR; Kira writes bilingual copy; Nyx still applies Saudi veto
- **Phase 6** (Challenge Round): each pre-mortem failure mode must be explicitly addressed; if a risk is left unmitigated, deliverable flags `⚠ UNMITIGATED PRE-MORTEM RISK`

## Decision Log (E3) — full schema

**New file:** `forge-decisions.json` at project root, mirrored atomically to `assets/forge-decisions.json`.

**Schema:**
```json
{
  "decisions": [
    {
      "id": "dec-<8 lowercase hex>",
      "title": "Short imperative — verb + object",
      "context": "What was the situation that required this decision",
      "alternatives_considered": [
        "Option A (reject: reason)",
        "Option B (reject: reason)",
        "Option C (selected)"
      ],
      "decided_by": "agent-<id>",
      "dissenting": ["agent-<id>", ...],
      "dissent_reason": "why the dissenter disagreed",
      "decided_at": "ISO 8601 UTC",
      "reversibility": "type_1" | "type_2",
      "review_at": "ISO 8601 UTC",
      "project_id": "proj-<id>",
      "related_evidence": ["ev-<id>", ...],
      "status": "open" | "reviewed" | "reversed" | "committed"
    }
  ],
  "project_decision_index": {
    "proj-<id>": ["dec-<id>", ...]
  }
}
```

**Reversibility taxonomy (Amazon/Bezos):**
- **Type 1** — one-way door: hard to reverse. Cabinet vote required. Reviewed at 90d default.
- **Type 2** — two-way door: easy to reverse. IC can decide; escalates only on deadlock. Reviewed at 30d default.

**Status lifecycle:**
- `open` — decision stands, review not yet due
- `reviewed` — review happened, decision reaffirmed
- `reversed` — subsequent decision overrode this one (link to successor)
- `committed` — implemented and locked in; cannot be easily reversed regardless of original type

**New module:** `tools/decisions_orchestrator.py` with `append_decision`, `review_decisions_due`, `close_decision`, `reverse_decision`. Pattern matches `tools/evidence_orchestrator.py`:
- Atomic write to primary path
- Mirror to `assets/` sibling if present
- Same tempfile + os.replace idiom

**Validator extension:** `validate_decisions(doc, state, evidence_doc)` in `tools/validator.py`:
- `id` format `dec-[0-9a-f]{8}`
- `decided_by` + `dissenting` agents must exist
- `reversibility` ∈ {type_1, type_2}
- `status` ∈ {open, reviewed, reversed, committed}
- `related_evidence` IDs must exist in forge-evidence.json
- `project_decision_index` refs must point to existing decisions
- `decided_at` + `review_at` valid ISO 8601

**Protocol rule (Standing Rule 11 — new):** Every Cabinet Verdict in Phase 7 must log at least one Decision Log entry. Enforcement: validator flags project_evidence_index entries whose corresponding project lacks a `project_decision_index` entry.
